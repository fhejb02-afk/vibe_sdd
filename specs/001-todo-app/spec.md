# 기능 명세서: 오늘의 할 일 웹 앱

**Feature Branch**: `001-todo-app`

**Created**: 2026-07-15

**Status**: Draft

**Input**: 사용자 요구사항: “오늘의 할 일” 웹 앱을 만들고자 하며, 사용자는 할 일을 생성·완료·삭제하고 목록을 필터링하며, 데이터가 유지되고 모바일 화면에서도 사용할 수 있어야 한다.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 할 일을 추가하고 안내를 받기 (우선순위: P1)
사용자는 새 할 일을 입력하고 목록에 바로 추가할 수 있어야 한다. 입력값이 비어 있거나 공백만 있는 경우에는 항목이 추가되지 않고, 왜 추가되지 않았는지 알 수 있는 안내 메시지가 표시되어야 한다.

**Why this priority**: 할 일 관리의 기본 동작이므로 이 기능이 없으면 앱의 핵심 가치를 제공하지 못한다.

**Independent Test**: 사용자가 텍스트를 입력해 추가를 시도하고, 빈 입력에 대해 안내를 확인하는 흐름만으로 독립적으로 검증할 수 있다.

**수용 시나리오**:

1. **Given** 사용자가 할 일 목록이 비어 있는 상태에 있을 때, **When** 새 할 일을 입력하고 추가를 실행하면, **Then** 해당 할 일이 목록에 추가되고 남은 할 일 수가 즉시 갱신된다.
2. **Given** 사용자가 빈 값이나 공백만 입력한 상태일 때, **When** 추가를 실행하면, **Then** 새 항목은 추가되지 않고 사용자에게 안내 메시지가 표시된다.

---

### 사용자 스토리 2 - 할 일을 완료/취소하고 삭제하기 (우선순위: P1)
사용자는 할 일을 완료 상태로 전환하거나 다시 미완료 상태로 되돌릴 수 있어야 한다. 완료된 항목은 시각적으로 구분되어야 하며, 삭제는 실수 방지를 위해 확인 절차를 거쳐야 한다.

**Why this priority**: 완료 상태 관리와 삭제는 할 일 앱의 핵심 상호작용이며, 사용자 경험에 직접 영향을 준다.

**Independent Test**: 하나의 항목에 대해 완료·취소·삭제를 순차적으로 수행해 각 상태 변화가 올바르게 반영되는지 확인할 수 있다.

**수용 시나리오**:

1. **Given** 목록에 미완료 항목이 있을 때, **When** 사용자가 해당 항목을 완료 처리하면, **Then** 항목은 완료 상태로 표시되고 남은 할 일 수가 감소한다.
2. **Given** 완료된 항목이 있을 때, **When** 사용자가 해당 항목을 다시 미완료로 전환하면, **Then** 항목은 일반 상태로 복원되고 남은 할 일 수가 다시 증가한다.
3. **Given** 삭제하고자 하는 항목이 있을 때, **When** 사용자가 삭제를 시도하면, **Then** 삭제 전 확인 절차가 나타나고 확인 후에만 항목이 제거된다.

---

### 사용자 스토리 3 - 목록을 필터링하고 빈 상태를 인지하기 (우선순위: P2)
사용자는 전체, 진행 중, 완료된 항목으로 목록을 필터링할 수 있어야 한다. 할 일이 하나도 없을 때는 명확한 안내 문구를 볼 수 있어야 한다.

**Why this priority**: 필터와 빈 상태는 사용자가 목록을 쉽게 파악하고 다음 행동을 결정하는 데 도움이 되는 필수 UX 요소다.

**Independent Test**: 필터를 변경하고 할 일을 모두 비운 뒤 빈 상태 메시지가 표시되는 흐름으로 검증할 수 있다.

**수용 시나리오**:

1. **Given** 여러 상태의 할 일이 있을 때, **When** 사용자가 “진행중” 필터를 선택하면, **Then** 완료된 항목은 보이지 않고 진행 중 항목만 표시된다.
2. **Given** 할 일이 하나도 없을 때, **When** 목록이 렌더링되면, **Then** 빈 상태 안내 문구가 화면에 표시된다.

---

### 엣지 케이스

- 빈 값이나 공백만 있는 입력이 제출될 때 어떻게 처리되는가?
- 삭제 확인 절차에서 사용자가 취소하면 항목이 그대로 유지되는가?
- 모든 할 일을 삭제한 뒤 남은 개수와 빈 상태 안내가 올바르게 표시되는가?
- 화면 폭이 360px로 좁아졌을 때 주요 조작이 가로로 압축되지 않는가?

## 요구사항 *(필수)*

### 결정된 구현 방향

- **DR-001**: 데이터 영속성은 브라우저의 localStorage를 사용하여 구현한다. 새로고침이나 브라우저 재시작 후에도 할 일 목록이 유지된다.
- **DR-002**: 구현 범위는 웹 앱의 기본 할 일 관리 기능에 한정하며, 서버 기반 백엔드나 데이터베이스는 포함하지 않는다.

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 텍스트 입력을 통해 새 할 일을 추가할 수 있어야 한다.
- **FR-002**: 시스템은 빈 값이나 공백만 있는 입력을 거부하고, 사용자에게 안내 메시지를 표시해야 한다.
- **FR-003**: 시스템은 사용자가 할 일을 완료 상태로 전환하고 다시 미완료 상태로 되돌릴 수 있어야 한다.
- **FR-004**: 시스템은 완료된 항목을 시각적으로 구분하여 표시해야 하며, 완료된 항목은 취소선과 흐린 색으로 나타나야 한다.
- **FR-005**: 시스템은 사용자가 할 일을 삭제할 수 있도록 해야 하며, 삭제 전 확인 절차를 제공해야 한다.
- **FR-006**: 시스템은 화면 상단에 남은 할 일 개수를 항상 표시해야 하며, 추가·완료·삭제 동작이 발생할 때 즉시 갱신되어야 한다.
- **FR-007**: 시스템은 전체, 진행 중, 완료 3가지 필터를 제공해야 한다.
- **FR-008**: 시스템은 브라우저 새로고침과 서버 재시작 후에도 할 일 목록이 유지되도록 저장해야 한다.
- **FR-009**: 시스템은 360px 폭의 모바일 화면에서도 사용할 수 있도록 반응형 레이아웃을 제공해야 한다.
- **FR-010**: 시스템은 할 일이 하나도 없을 때 친절한 빈 상태 안내 문구를 표시해야 한다.

### 핵심 엔티티 *(데이터가 관련된 기능이므로 포함)*

- **할일 항목(Todo Item)**: 사용자가 관리하는 할 일의 기본 단위로, 내용, 완료 여부, 생성 시점 정보를 포함한다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 사용자는 주요 동작(추가, 완료/취소, 삭제, 필터링)을 3분 이내에 성공적으로 수행할 수 있어야 한다.
- **SC-002**: 사용자가 새로고침하거나 서버를 재시작한 후에도 저장된 할 일 목록이 유지되어야 하며, 누락 없이 확인할 수 있어야 한다.
- **SC-003**: 360px 폭 화면에서도 주요 입력, 필터, 상태 표시 조작이 모두 접근 가능해야 한다.
- **SC-004**: 할 일이 없을 때 빈 상태 안내가 표시되어 사용자가 다음 행동을 이해할 수 있어야 한다.
- **SC-005**: 사용자는 필터와 남은 개수 정보를 보고 현재 목록의 상태를 즉시 파악할 수 있어야 한다.

## 가정

- 초기 버전은 로그인 없이 단일 사용자를 기준으로 동작하는 것으로 가정한다.
- 데이터 영속성은 사용자의 브라우저나 서비스 환경에서 안정적으로 유지되는 것을 목표로 한다.
- 모바일 반응형 지원은 360px 폭의 화면에서 핵심 기능이 모두 사용 가능함을 의미한다.
